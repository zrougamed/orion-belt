How Reverse SSH Tunneling Works in Orion-Belt

January 8, 2026 · 2 min read
Mohamed Zrouga
Creator of Orion-Belt

One of the key features of Orion-Belt is its use of reverse SSH tunnels to establish connections. This architectural choice eliminates the need for inbound firewall rules on target machines, but how does it actually work?

The Problem with Traditional SSH Access​

Traditional SSH requires opening port 22 (or another SSH port) on each target machine to accept incoming connections. This creates several challenges:

  • Firewall management becomes complex as the fleet grows
  • Each exposed port is a potential attack surface
  • Dynamic IPs and NAT complicate access
  • Monitoring and logging are fragmented across machines

Reverse Tunneling Approach​

Orion-Belt inverts this model. Instead of clients connecting directly to target machines, the target machines connect outbound to the gateway server. When a user wants to SSH into a machine, the gateway routes the connection through the existing tunnel.

Here's the flow:

  1. Agent on target machine establishes persistent SSH connection to gateway
  2. Agent registers itself and maintains the connection
  3. User authenticates with gateway and requests access to target
  4. Gateway verifies permissions and routes SSH session through existing tunnel
  5. User's session appears as a normal SSH connection to target machine

Benefits​

This architecture provides several advantages:

Security: Target machines never listen for incoming connections. Only one component (the gateway) needs to be internet-accessible.

Simplicity: No firewall rules needed on target machines. Just one outbound connection.

Observability: All access flows through a single point, making audit logs comprehensive and consistent.

Flexibility: Works behind NAT, with dynamic IPs, and across complex network topologies.

Implementation Details​

The agent uses Go's golang.org/x/crypto/ssh package to create the reverse tunnel. The connection includes automatic reconnection logic and heartbeat mechanisms to maintain connectivity.

On the gateway side, the SSH proxy maintains a registry of active tunnels and routes incoming user sessions to the appropriate agent connection. Session data passes through the gateway, enabling real-time monitoring and recording without compromising end-to-end encryption of the SSH protocol itself.

Trade-offs​

This approach does introduce a single point of failure (the gateway), which is why high availability features are planned for v1.0. The gateway must also handle the multiplexing of potentially hundreds of concurrent sessions, requiring careful resource management.

For most deployments, these trade-offs are acceptable given the security and operational benefits.
